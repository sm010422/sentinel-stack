# Layer 2: 커스텀 통신 프로토콜 명세 (sentinel Protocol v1)

> 버전: 1.0 | 참고: Link-16, MIL-STD-1553, DDS OMG 표준

---

## 1. 개요

sentinel 프로토콜은 전술 통신 환경의 요구사항을 반영한 커스텀 바이너리 프로토콜입니다.

**설계 원칙**
- 최소 헤더 오버헤드 (38 bytes)
- 강한 보안: AES-256-GCM 인증 암호화
- 전송 오류 감지: CRC-32
- 토픽 기반 라우팅 (DDS 참고)
- TCP + UDP 이중화 채널

---

## 3. 필드 정의

| 필드 | 크기 | 설명 |
|------|------|------|
| Magic | 4B | 0x53454E54 ("SENT") — 프로토콜 식별 |
| Version | 1B | 프로토콜 버전 (현재 1) |
| Msg Type | 1B | 0x01=DATA, 0x02=STATUS, 0x03=CMD, 0x04=HB, 0x05=ACK |
| Seq Num | 4B | 발신 노드 기준 증가 시퀀스 번호 |
| Timestamp | 8B | Unix 타임스탬프 (밀리초) |
| Src Node | 1B | 발신 노드 ID (1~254) |
| Dst Node | 1B | 수신 노드 ID (0xFF=브로드캐스트) |
| Topic ID | 2B | 0x0001=SENSOR, 0x0002=STATUS, 0x0003=CMD, 0x0004=HB |
| Payload Len | 4B | 암호화된 페이로드 길이 |
| AES IV | 12B | AES-256-GCM 초기화 벡터 (매 패킷 무작위) |
| Enc Payload | 가변 | AES-256-GCM 암호화 페이로드 |
| AES Tag | 16B | GCM 인증 태그 |
| CRC-32 | 4B | 헤더+페이로드+태그에 대한 CRC (IEEE 802.3) |

---

## 4. 보안 설계

### AES-256-GCM 인증 암호화

- **기밀성**: AES-256 CBC 대신 GCM 모드로 스트림 암호화
- **무결성**: 128비트 GCM 태그로 변조 감지
- **IV 관리**: 매 패킷 `/dev/urandom` 기반 12바이트 무작위 생성  
  ⚠️ IV 재사용 시 GCM 보안 완전 붕괴 — 재사용 절대 금지

### CRC-32

- 전송 오류(비트 플립) 감지용
- 보안 목적이 아님 (GCM 태그가 무결성 담당)

---

## 5. 채널 이중화 (MIL-STD-1553 참고)

| 채널 | 포트 | 용도 | 특성 |
|------|------|------|------|
| TCP 9000 | 9000 | CMD, STATUS, 중요 데이터 | 신뢰성, 재전송 |
| UDP 9001 | 9001 | SENSOR_DATA, HEARTBEAT | 저지연, 무손실 불보장 |

**Heartbeat 기반 장애 감지**
- 1000ms 주기로 HEARTBEAT 패킷 전송
- 3회 연속 무응답 → 연결 끊김 판단
- 500ms 이내 자동 재연결

## 6. Publisher-Subscriber 라우팅

DDS(Data Distribution Service, OMG) 개념 참고:

```
Publisher (Node #1)          Subscriber (Node #2)
─────────────────            ───────────────────
pubsub_publish()             pubsub_subscribe()
    │                            │
    ▼                            ▼
[SENSOR_DATA] ────────────── [SENSOR_DATA] callback()
[STATUS]      ────────────── [STATUS]      callback()
```
